### Sequence Parallelism

`Sequence Parallelism` 沿着序列长度维度切数据。也就是说，不同 GPU 负责同一个 batch 里的不同 token 片段。

```text
tokens: [t0 t1 t2 t3 t4 t5 t6 t7]

GPU0: [t0 t1]
GPU1: [t2 t3]
GPU2: [t4 t5]
GPU3: [t6 t7]
```

它常和 tensor parallelism 配合使用，主要目标是降低每张 GPU 上的 activation 显存压力。序列越长，activation 和中间状态越容易成为瓶颈。

### Activation

#### 为什么按序列切

Transformer 里有些计算的参数量不大，但 activation 会跟序列长度一起增长，比如 layer norm、dropout、残差连接相关的中间张量。

如果所有 GPU 都保留完整序列的 activation，就会产生重复存储。Sequence parallelism 把这些 token 维度上的 activation 分摊到多张 GPU 上：

$$
M_{act}^{gpu}
\approx
\frac{B \times L \times D}{G}
$$

其中 `B` 是 batch size，`L` 是 sequence length，`D` 是 hidden dimension，`G` 是 GPU 数量。

这样每张卡上保留的序列片段更短，显存压力也更低。

### Attention

#### 不能随便切断依赖

Sequence parallelism 的难点在 attention。对一个 query token 来说，它可能需要看同一序列里所有历史 token 的 `K/V`。如果序列被切到不同 GPU 上，就要通过通信交换或聚合必要信息。

所以 sequence parallelism 更常用于那些可以沿 token 维度局部处理的部分；遇到 attention 这种全局依赖时，需要和 tensor parallelism、通信 collective 或专门的 attention kernel 配合。

### Inference

#### 长上下文下更有意义

在推理里，sequence parallelism 对长 prompt 的 `Prefill` 更有价值，因为此时一次要处理很多 token，activation 和 attention 中间状态都比较大。

到了 `Decode` 阶段，每一步通常只新增一个 token，序列维度上的并行空间变小，瓶颈更多转向 KV Cache 读取、跨 GPU 通信和调度。

### Summary

Sequence Parallelism 是按 token 维度切序列。它适合缓解长序列带来的 activation 显存压力，但 attention 的全局依赖会引入额外通信，因此通常要和其他并行方式一起设计。
