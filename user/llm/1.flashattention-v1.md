### 核心问题

#### Attention 符号速览
Transformer 里的 self-attention 可以先按下面三步理解：

$$
S = QK^T,
\qquad
P = \operatorname{softmax}(S),
\qquad
O = P V
$$

- `Q`：Query，表示当前 token 想要查询什么信息。
- `K`：Key，表示每个 token 可以被匹配的特征。
- `V`：Value，表示每个 token 真正要贡献出去的内容向量。
- `S = QK^T`：计算 token 之间的匹配分数。
- `P = softmax(S)`：把匹配分数变成注意力权重。
- `P @ V` 或 `P V`：矩阵乘法，用注意力权重 `P` 对 `V` 做加权求和，得到输出 `O`。

通俗地说，`P` 决定“该看谁、看多少”，`V` 是“被看的内容”，`P @ V` 就是“按权重把内容混合起来”。FlashAttention 优化的正是 `S -> P -> O` 这条链路里大量中间矩阵的读写。

#### 标准 Attention 的瓶颈
- 背景：Transformer 里的自注意力通常写成 `S = QK^T / sqrt(d)`、`P = softmax(S)`、`O = PV`。
- 问题：`S` 和 `P` 都是序列长度平方级别的矩阵，长序列下会占用大量 HBM 显存读写带宽。
- 关键观察：很多 Attention kernel 的瓶颈不是算力，而是把中间矩阵反复写入和读出 HBM。
- FlashAttention-V1 的目标：在保持 exact attention 结果的前提下，减少 HBM 访问，不近似、不稀疏化、不改变模型数学定义。

标准实现为了方便框架调度，会把 `QK^T`、softmax 结果和 dropout/mask 之后的概率矩阵显式物化出来。这样代码清楚，但会产生大量中间读写。FlashAttention-V1 的核心思路是把这些中间状态留在 SRAM/shared memory 和寄存器里，以 block 为单位边算边归约。

在 MHA profiling 里，scaled dot-product attention 经常很耗时。原因不是 `QK^T` 这次矩阵乘本身特别难，而是 `QK^T` 的形状是 `[batch_size, num_heads, seqlen, seqlen]`。`seqlen` 一变长，这个中间矩阵就按平方级膨胀；前向要写，softmax 要读写，反向还可能为了算梯度再读一次。这就是“显存流量”成为瓶颈的根源。

### IO-aware 思路

#### 从伪融合到真正融合
- 早期很多 fused attention 更像是“把 matmul、scale、mask、softmax、matmul 的调用封装到一个接口里”。
- 这种封装能减少框架层的调度开销，但如果中间的 `S` 和 `P` 仍然落到 HBM，就没有真正解决 memory traffic。
- FlashAttention-V1 的融合更彻底：它把 score、softmax 和 `P @ V` 放进同一个分块流程里，尽量让中间结果只在片上存储中短暂停留。
- 通俗地说，伪融合像是把多个窗口排到一个柜台后面，流程还是一站一站搬材料；真正融合是把材料放在同一张工作台上连续加工。

#### HBM 与 SRAM 的角色
- HBM：容量大但访问慢，适合存放完整的 `Q`、`K`、`V` 和最终输出 `O`。
- SRAM/shared memory：容量小但访问快，适合缓存当前 block 的 `Q`、`K`、`V` tile。
- 寄存器：保存每一行 softmax 的运行最大值、归一化分母和局部输出累加。
- 设计重点：让每个 `K/V` tile 被加载后尽可能多地复用，避免完整 attention score 矩阵落回 HBM。

FlashAttention-V1 不是减少乘加次数，而是减少 IO。它仍然计算完整的 attention，只是把计算顺序改成分块扫描：每次取一块 `Q` 和一块 `K/V`，完成局部 score、局部 softmax 统计和输出累加，然后继续处理下一块。

#### 为什么能省显存
- 标准 Attention 会显式保存 `N x N` 的 score/probability 矩阵。
- FlashAttention-V1 不保存完整 `N x N` 中间矩阵，只保存每行的少量统计量。
- 前向阶段通常只需要保存输出、softmax 的归一化统计，以及反向需要重算的必要信息。
- 显存节省来自“少存中间矩阵”，速度提升来自“少访问慢速 HBM”。

这里最大的难点是 softmax。softmax 是 row-wise 操作：一行里每个元素的结果，都依赖这一整行的最大值和分母。如果按普通写法，必须先把一整行 score 算完并保存下来，后面的 `P @ V` 才能继续。FlashAttention-V1 的做法是维护每行的在线统计量，让“一整行”可以被拆成多个小块陆续处理。

### 分块算法

分块计算的关键不只是把 `QK^T` 切成更小的矩阵乘。矩阵乘的局部结果可以按块累加，softmax 却不能直接这样拆开：同一行里的每个概率都依赖整行最大值和整行分母。如果每个小块各自归一化，再把结果拼起来，得到的就不再是全局 softmax。

因此，FlashAttention 需要把 softmax 的全行依赖改写成可以随扫描过程更新的状态。本节从普通 Softmax 出发，先引入用于数值稳定的 Safe Softmax，再用 Online Softmax 合并最大值和分母的更新；随后把这组在线状态接入 Self-Attention 的 `V` 加权累加，得到 FlashAttention 的单行递推，最后扩展为按 tile 扫描的块级算法。整个过程的目标是保持 exact attention 的数学结果，同时避免把完整的 `N x N` score/probability 矩阵写入 HBM。

#### Softmax
Softmax 的作用是把一行 score `x` 变成一行概率 `a`。先用最普通的 softmax 写法引出两个状态：输出 `a_i` 和分母 `d_i`。对一行输入：

$$
x = [x_1,\ x_2,\ ...,\ x_N]
$$

第 `i` 个位置的输出是：

$$
a_i =
\frac{e^{x_i}}
{\sum_{j=1}^{N} e^{x_j}}
$$

为了后面能和 Online Softmax 对齐，把分母写成前缀累计：

$$
d_i = d_{i-1} + e^{x_i},
\qquad
d_0 = 0
$$

扫完整行后得到：

$$
d_N = \sum_{j=1}^{N}e^{x_j}
$$

第二遍再输出：

$$
a_i = \frac{e^{x_i}}{d_N}
$$

因此普通 softmax 可以理解成一个 2-pass 算法：

1. Pass 1：扫描整行，累计分母 `d_N`。
2. Pass 2：再次扫描整行，输出 `a_i = exp(x_i) / d_N`。

这里的关键限制是：`a_i` 依赖整行最终分母 `d_N`。也就是说，`a_i` 不是只由 `x_i` 决定，而是依赖整行 `x`。这正是 softmax 很难直接分块的原因：如果只看到当前 block，就不知道最终分母是多少。

把它放到 attention 里，一行输出可以写成：

$$
o =
\sum_{i=1}^{N} a_i v_i
=
\frac{\sum_{i=1}^{N} e^{x_i}v_i}
{\sum_{i=1}^{N} e^{x_i}}
$$

其中 `x_i = qK_i^T`，`v_i` 是对应的 value 向量。标准实现通常先得到整行 `x`，再计算 `a`，最后用 `a` 乘 `V`。这很直观，但在长序列下会迫使 `QK^T` 和 softmax 后的概率矩阵占用大量显存。

#### Safe Softmax
普通 softmax 的问题是 `e^{x_i}` 可能溢出。Safe Softmax 用一个数学等价的形式解决数值稳定性：先找整行最大值，再把所有元素减去这个最大值。

第一遍扫描整行，得到前缀最大值：

$$
m_i = \max(m_{i-1}, x_i),
\qquad
m_0 = -\infty
$$

扫完整行后：

$$
m_N = \max_{1 \le j \le N} x_j
$$

第二遍在最终最大值 `m_N` 的尺度下累计分母：

$$
d_i = d_{i-1} + e^{x_i - m_N},
\qquad
d_0 = 0
$$

扫完整行后：

$$
d_N = \sum_{j=1}^{N} e^{x_j-m_N}
$$

第三遍输出概率：

$$
a_i =
\frac{e^{x_i - m_N}}
{d_N}
$$

这个变换不改变 softmax 结果，因为分子和分母都除以了同一个 `e^{m_N}`：

$$
\frac{e^{x_i-m_N}}
{\sum_j e^{x_j-m_N}}
=
\frac{e^{x_i}/e^{m_N}}
{\sum_j e^{x_j}/e^{m_N}}
=
\frac{e^{x_i}}
{\sum_j e^{x_j}}
$$

Safe Softmax 的好处是数值稳定：因为 `x_i - m_N <= 0`，指数不会因为过大的正数溢出。但它带来一个重要代价：如果输入行不能完整留在 SRAM 里，算法会变成 3-pass；放到 self-attention 里，也就意味着同一行 logits 可能要通过 `Q` 和 `K` 重算多次。

1. Pass 1：扫描整行，得到最终最大值 `m_N`。
2. Pass 2：再次扫描整行，得到稳定化分母 `d_N = sum_j exp(x_j - m_N)`。
3. Pass 3：第三次扫描整行，输出 `a_i = exp(x_i - m_N) / d_N`。

关键问题在第二遍：`d_i` 的定义依赖最终最大值 `m_N`。在第一遍还没扫完整行之前，局部 block 只能知道当前最大值 `m_i`，不能直接把分母累计到最终尺度上。Online Softmax 要解决的就是这个尺度切换问题。

#### Online Softmax
Online Softmax 要解决的是 Safe Softmax 的一个具体问题：第二遍计算分母时需要最终最大值 `m_N`，但 `m_N` 只有第一遍结束后才知道。解决办法是引入一个随前缀最大值变化的代理分母 `d'_i`。

先定义前缀最大值：

$$
m_i = \max(m_{i-1}, x_i)
$$

再定义代理分母：

$$
d'_i = \sum_{j=1}^{i} e^{x_j - m_i}
$$

这里的关键是，`d'_i` 用的是当前前缀最大值 `m_i`，不是最终最大值 `m_N`。当 `i = N` 时，`m_i` 正好等于 `m_N`，所以：

$$
d'_N =
\sum_{j=1}^{N} e^{x_j-m_N}
=
d_N
$$

也就是说，只要能在扫描时递推 `d'_i`，最后就能得到 Safe Softmax 需要的同一个分母。

推导 `d'_i` 和 `d'_{i-1}` 的关系时，把第 `i` 项单独拆出来：

$$
d'_i
=
\sum_{j=1}^{i} e^{x_j-m_i}
=
\sum_{j=1}^{i-1} e^{x_j-m_i}
+
e^{x_i-m_i}
$$

前 `i-1` 项原来是在 `m_{i-1}` 这个尺度下累计的。要换到新的 `m_i` 尺度，只需要乘上一个缩放因子：

$$
d'_i
=
\left(
\sum_{j=1}^{i-1} e^{x_j-m_{i-1}}
\right)
e^{m_{i-1}-m_i}
+
e^{x_i-m_i}
$$

于是得到 Online Softmax 的核心递推：

$$
d'_i =
d'_{i-1}e^{m_{i-1}-m_i}
+
e^{x_i-m_i}
$$

这条式子只依赖当前元素 `x_i`、旧状态 `m_{i-1}/d'_{i-1}` 和新最大值 `m_i`，因此最大值和分母可以在同一遍扫描里更新。

2-pass online softmax 可以写成两个循环：

1. 第一遍扫描输入，同时更新：

$$
m_i \leftarrow \max(m_{i-1}, x_i)
$$

$$
d'_i \leftarrow
d'_{i-1}e^{m_{i-1}-m_i}
+
e^{x_i-m_i}
$$

2. 第二遍扫描输入，输出 softmax 概率：

$$
a_i \leftarrow
\frac{e^{x_i-m_N}}{d'_N}
$$

初始化可以理解为 `m_0 = -\infty`、`d'_0 = 0`。第一遍结束后，`m_N` 是整行最大值，`d'_N` 是稳定 softmax 的最终分母；第二遍只负责把每个 `x_i` 归一化成概率 `a_i`。

和 Safe Softmax 相比，Online Softmax 把“求最大值”和“求分母”合并到第一遍，因此从 3-pass 变成 2-pass。但如果目标只是输出 softmax 概率，仍然需要第二遍，因为每个 `a_i` 依赖最终的 `m_N / d'_N`。它也还没有把 softmax 输出和后面的 `V` 加权累加融合起来：如果把整行概率写成 `A = softmax(QK^T)` 再去乘 `V`，仍然会有完整 `N x N` 中间矩阵的问题。

这正是下一节 Multi-pass Self-Attention 要处理的问题：能不能在 attention 的输出递推里复用同样的 `m_i/d'_i` 状态，而不是先完整产出概率矩阵。

#### Multi-pass Self-Attention
Online Softmax 已经把 Safe Softmax 的前两遍合并了，但 Self-Attention 的目标不是输出概率行 `a`，而是输出：

$$
O = \operatorname{softmax}(QK^T)V
$$

先固定输出矩阵的一行来描述算法。设当前计算第 `k` 个 query，对 batch/head、scale 和 mask 暂时忽略，只关注一行的递推：

- `Q[k, :]`：query 矩阵的第 `k` 行。
- `K^T[:, i]`：第 `i` 个 key 向量。
- `V[i, :]`：第 `i` 个 value 向量。
- `O[k, :]`：最终要写出的第 `k` 行输出。
- `x_i`：当前 query 和第 `i` 个 key 的 dot product。
- `a_i`：当前 query 对第 `i` 个 token 的 attention weight。
- `o_i`：只累加到前 `i` 个 value 后的部分输出。

第一遍循环只做 Online Softmax 的统计量，不产生最终输出：

$$
x_i = Q[k,:]K^T[:,i]
$$

$$
m_i = \max(m_{i-1}, x_i)
$$

$$
d'_i =
d'_{i-1}e^{m_{i-1}-m_i}
+
e^{x_i-m_i}
$$

扫完整行后，`m_N` 和 `d'_N` 才是最终 softmax 所需的归一化状态。第二遍循环再重新扫描 key/value，计算概率和输出：

$$
a_i =
\frac{e^{x_i-m_N}}{d'_N}
$$

$$
o_i = o_{i-1} + a_iV[i,:]
$$

最后得到：

$$
O[k,:] = o_N
$$

这就是 Multi-pass 的含义：虽然没有显式保存完整的 `A = softmax(QK^T)`，但仍然需要先完成第一遍，拿到最终 `m_N` 和 `d'_N`，第二遍才能计算 `a_i` 并累加 `o_i`。原因是 `a_i` 依赖全行最终归一化状态：

$$
a_i =
\frac{e^{x_i-m_N}}{d'_N}
$$

把 `a_i` 直接代回输出递推，可以看到问题在哪里：

$$
o_i =
o_{i-1}
+
\frac{e^{x_i-m_N}}{d'_N}V[i,:]
$$

右侧仍然依赖 `m_N` 和 `d'_N`，而这两个量只有第一遍扫完整行之后才知道。因此 Multi-pass Self-Attention 还不能把 score、softmax 和 `V` 的加权累加放进同一个循环。

接下来做的变换和 Online Softmax 很像：不要直接维护依赖最终 `m_N/d'_N` 的 `o_i`，而是构造一个随当前前缀状态变化的代理输出：

$$
o'_i =
\sum_{j=1}^{i}
\frac{e^{x_j-m_i}}{d'_i}V[j,:]
$$

它和最终输出的关系是：

$$
o'_N = o_N
$$

因为扫到最后时 `m_i = m_N` 且 `d'_i = d'_N`。关键是 `o'_i` 只依赖当前前缀状态，可以写成递推形式：

$$
o'_i =
\frac{d'_{i-1}e^{m_{i-1}-m_i}}{d'_i}o'_{i-1}
+
\frac{e^{x_i-m_i}}{d'_i}V[i,:]
$$

这一步说明了 FlashAttention 为什么可能做到单次扫描：如果同时维护 `m_i`、`d'_i` 和 `o'_i`，每来一个新的 key/value，就可以立刻把它合并到当前输出状态里，而不必等到最终 `m_N/d'_N` 已知后再回头扫第二遍。

#### FlashAttention
上一节已经构造出代理输出 `o'_i`，并且它满足两个条件：

- `o'_i` 只依赖当前前缀状态 `m_i / d'_i`，不依赖最终的 `m_N / d'_N`。
- 扫完整行后，`o'_N = o_N`，也就是代理输出的最后一项就是 Self-Attention 的真实输出。

因此 FlashAttention 可以把 Online Softmax 的状态 `(m_i, d'_i)` 和输出状态 `o'_i` 放进同一个循环。对第 `k` 个 query 行，初始化 `m_0 = -\infty`、`d'_0 = 0`、`o'_0 = 0`，单行算法可以写成：

$$
x_i \leftarrow Q[k,:]K^T[:,i]
$$

$$
m_i \leftarrow \max(m_{i-1}, x_i)
$$

$$
d'_i \leftarrow
d'_{i-1}e^{m_{i-1}-m_i}
+
e^{x_i-m_i}
$$

$$
o'_i \leftarrow
o'_{i-1}
\frac{d'_{i-1}e^{m_{i-1}-m_i}}{d'_i}
+
\frac{e^{x_i-m_i}}{d'_i}V[i,:]
$$

循环结束后：

$$
O[k,:] \leftarrow o'_N
$$

这条递推可以直接从上一节的 `o'_i` 公式读出来：

$$
o'_i =
\frac{d'_{i-1}e^{m_{i-1}-m_i}}{d'_i}o'_{i-1}
+
\frac{e^{x_i-m_i}}{d'_i}V[i,:]
$$

第一项表示“旧输出换尺度”：旧的 `o'_{i-1}` 原来除的是 `d'_{i-1}`，并且处在 `m_{i-1}` 的指数尺度下；现在最大值和分母变成 `m_i / d'_i`，所以要乘上：

$$
\frac{d'_{i-1}e^{m_{i-1}-m_i}}{d'_i}
$$

第二项是当前 `V[i,:]` 的新贡献：

$$
\frac{e^{x_i-m_i}}{d'_i}V[i,:]
$$

这样，每看到一个新的 key/value，就完成三件事：更新最大值、更新分母、更新输出。整行不再需要先生成 `A[k,:] = softmax(Q[k,:]K^T)`，也不需要第二遍回头把概率乘以 `V`。

这里也解释了一个容易混淆的点，讲得直白一点：Online Softmax 的第一遍其实只是“把账记完”。它先把整行的最大值和分母算出来，等这两个全局量确定后，第二遍才能把每个位置真正换算成 softmax 概率。所以它虽然把“找最大值”和“累分母”合并了，最后还是要再扫一遍来产出概率。

FlashAttention 走得更远一点。它不先把整行概率全部算出来，而是直接盯着最终输出。每扫到一个新的 key/value，就顺手把这部分 value 对输出的贡献加进去，旧输出也一起按新的 softmax 尺度调整好。这样只关心最终输出时，每个 key/value 看一次就够；如果还想显式写出完整概率矩阵，那自然又要额外多走一遍。

这个版本仍然是“单行、逐元素”的说明。它展示的是 FlashAttention 的核心递推，而不是最终 GPU kernel 的形态。实际 GPU 上会把多个 `x_i` 合成一个 tile 向量来处理，并把 `Q/K/V/O/m/d'` 的状态放到 SRAM/shared memory 和寄存器中。下一节的 tiling 版本就是把上面的逐元素递推改写成块级递推。

#### FlashAttention (Tiling)
上面的 FlashAttention 递推仍然是逐元素扫描。tiling 版本先固定第 `k` 个 query 行，然后把这一行要扫描的 `K/V` 序列按 tile 切开。这里先讲单行按 tile 扫描的形式，再在下一章过渡到工程 kernel 里的多 query 行并行处理。

新记号如下：

- `b`：每个 tile 的 block size。
- `#tiles`：一行中的 tile 数量，满足 `N = b \times #tiles`。
- `x_i`：第 `i` 个 tile 的 score 向量，长度为 `b`。
- `m_i^{\text{local}}`：第 `i` 个 tile 内部的局部最大值。

状态仍然初始化为 `m_0 = -\infty`、`d'_0 = 0`、`o'_0 = 0`。

第 `i` 个 tile 对应 key/value 的范围是：

$$
[(i-1)b : ib]
$$

因此局部 score 向量是：

$$
x_i
\leftarrow
Q[k,:]K^T[:,(i-1)b:ib]
$$

这里的 `x_i[j]` 表示当前 query 行和这个 tile 内第 `j` 个 key 的 dot product。先在 tile 内求局部最大值：

$$
m_i^{\text{local}}
=
\max_{j=1}^{b}(x_i[j])
$$

再把局部最大值和历史最大值合并：

$$
m_i
\leftarrow
\max(m_{i-1},\ m_i^{\text{local}})
$$

分母递推也从逐元素版本变成 tile 版本。历史分母先换到新最大值尺度，再加上当前 tile 内所有元素的指数贡献：

$$
d'_i
\leftarrow
d'_{i-1}e^{m_{i-1}-m_i}
+
\sum_{j=1}^{b} e^{x_i[j]-m_i}
$$

输出递推同样按 tile 聚合。旧输出先按新的 softmax 尺度缩放，当前 tile 内每个 `V` 行再按对应权重加进来：

$$
o'_i
\leftarrow
o'_{i-1}
\frac{d'_{i-1}e^{m_{i-1}-m_i}}{d'_i}
+
\sum_{j=1}^{b}
\frac{e^{x_i[j]-m_i}}{d'_i}
V[j+(i-1)b,:]
$$

循环结构可以写成：

1. 对 `i = 1, ..., #tiles`：
2. 计算 `x_i = Q[k,:]K^T[:,(i-1)b:ib]`。
3. 计算 `m_i^{\text{local}} = max_j(x_i[j])`。
4. 更新 `m_i = max(m_{i-1}, m_i^{\text{local}})`。
5. 更新 `d'_i`。
6. 更新 `o'_i`。

扫完所有 tile 后：

$$
O[k,:]
\leftarrow
o'_{\#\text{tiles}}
$$

也可以写成 `o'_{N/b}`，因为 `#tiles = N/b`。

这就是 tiling 版本的重点：`x_i` 不再是一个标量，而是一个 tile 内的 score 向量；`m_i^{\text{local}}`、`d'_i` 和 `o'_i` 把这个 tile 的贡献一次性合并进当前行状态。完整 attention 概率行 `A[k,:]` 不需要物化，只需要在每个 tile 内短暂计算局部指数权重。

从硬件视角看：

- 当前参与计算的 `Q` block、`K/V` tile 和行级状态会放在 SRAM/shared memory 或寄存器中。
- 对 `K^T` 和局部 score，计算沿序列方向逐 tile 扫描。
- 对 `V`，按同样的 tile 顺序读取对应 value block。
- 状态 `m`、`d'` 和 `O` 随着 tile 扫描不断更新，只有必要的行级状态跨 tile 保留。

因为片上状态只和 `b`、`d` 有关，而不需要保存整行长度为 `N` 的 score/probability 矩阵，所以这个算法可以扩展到长序列而不被 `N \times N` 中间矩阵的 HBM 存储压垮。

### FlashAttention 流程

前面的推导已经把数学关系讲清楚了，这一节只看 kernel 里这些状态怎么流动。可以把它理解成三件事：前向时边扫边把 `m/d'/O` 留下来，反向时按 tile 重算局部 score，最后用更小的行级状态换掉整行概率矩阵。

#### 前向扫描
FlashAttention-V1 的前向可以理解成一层沿序列方向扫过去的循环：每次拿进一个 `K/V` tile，就用它去更新当前 `Q/O` block 的行级状态。局部 score、局部 softmax 和局部 `V` 累加都在片上完成，算完就丢；真正保留下来的只有还能继续往下接着算的 `O`、`m` 和 `d'`。

初始化很简单：

- `O = 0`，说明这行还没累加任何 `V`。
- `d' = 0`，说明分母还没开始积累。
- `m = -\infty`，说明当前还没有见过有效的最大值。

所以前向阶段的重点不是“先算完什么再算什么”，而是“每来一块就顺手更新一块”。这样 `S` 和 `P` 不会在 HBM 里待很久，片上只保留每行必要的状态。

#### 前向保存内容
前向结束后，反向真正需要保存的也很少：

- 输出 `O`：反向里计算 softmax 梯度时会用到前向输出。
- 每行的归一化信息：`m` 和 `d'`，或者它们合成后的 `L = m + log d'`。
- mask / dropout 这类必须和前向一致的信息。

说白了，FlashAttention 保存的不是一整张大表，而是“把那张表重建回来”所需要的行级摘要。

#### 反向传播
反向也一样，不去读完整 `P`，而是重新加载对应的 `Q/K/V` tile，再用前向保存的行级状态把这块概率恢复出来。恢复出来之后，先算 `dV`，再沿着 softmax 的行级公式往回推 `dQ` 和 `dK`。每算完一个 tile 就把它放下，不需要让整行概率常驻显存。

#### 为什么重算划算
多做一点局部矩阵乘和指数计算，换掉的是 `N x N` 中间矩阵的读写。对长序列来说，搬运这些大矩阵比重算更贵，所以 FlashAttention 的收益主要来自少写、少读、少让中间结果落到 HBM。序列越长，这个差距越明显。

### 适用场景

#### 什么时候收益明显
- 长序列训练和推理：序列越长，`N x N` 中间矩阵越大，IO 优化越明显。
- head dimension 适配 kernel 的场景：常见的 64、128 等维度更容易跑到优化路径。
- A100/H100 等高带宽 GPU：FlashAttention 能更好地把 SRAM/shared memory 和 Tensor Core 协同起来。
- 需要 exact attention 的模型：相比近似 attention，它不改变 attention 数学结果，更容易替换标准实现。

#### 需要注意的边界
- 短序列或小 batch：kernel launch、调度和其他算子可能占主要时间，收益不一定明显。
- 非典型 mask 或 shape：如果 kernel 没有覆盖对应模式，可能回退到普通 attention。
- 端到端速度：Attention 变快不代表整个模型同比例变快，layernorm、MLP、通信和 KV cache 管理也会影响总耗时。
- 数值细节：虽然是 exact attention，混合精度、mask、dropout 和累加路径仍然需要和框架实现对齐。

### 工程用法

#### 融合前后对比
融合前，scaled dot-product attention 可以拆成 `matmul -> scale -> mask -> softmax -> matmul`。这些算子各自执行时，中间矩阵很容易写回 HBM。融合后，FlashAttention-V1 把它们组织成一个 IO-aware kernel：score 小块刚算出来，就立刻做 mask、online softmax 和 `V` 的加权累积。

这可以理解为“流水线换位置”：不是先把所有半成品堆到仓库，再让下一个工序取走；而是每个小块在同一个工位上尽量加工完，只把必要的最终结果和少量统计量留下。

#### PyTorch 调用形态
现代框架通常会在 `scaled_dot_product_attention` 或上层库中自动选择 FlashAttention 类 kernel。工程上更常见的做法不是手写 CUDA，而是保证 dtype、mask、head dimension 和设备满足 kernel 条件。

```python
# PyTorch 侧的典型调用形态。
# 在支持的 GPU、dtype 和 shape 下，后端可能选择 FlashAttention 类 kernel。

import torch
import torch.nn.functional as F

# q/k/v 常见形状为 [batch, heads, seq_len, head_dim]。
q = torch.randn(batch, heads, seq_len, head_dim, device="cuda", dtype=torch.float16)
k = torch.randn(batch, heads, seq_len, head_dim, device="cuda", dtype=torch.float16)
v = torch.randn(batch, heads, seq_len, head_dim, device="cuda", dtype=torch.float16)

# is_causal=True 表示自回归模型常见的 causal mask。
out = F.scaled_dot_product_attention(q, k, v, is_causal=True)
```

#### 与普通 Attention 的差异
```python
# 普通写法会显式产生 scores 和 probs 这两个 N x N 中间矩阵。
scores = q @ k.transpose(-2, -1) * scale
probs = torch.softmax(scores, dim=-1)
out = probs @ v

# FlashAttention 的 kernel 会把上面三步融合到分块扫描里。
# 关键不是换一个 API，而是 kernel 内部不再物化完整 scores/probs。
```

#### CUDA 手搓 FA1 示例
下面是一个教学版 CUDA kernel，用来展示 FA1 的核心状态更新。它没有调用 FlashAttention 库，也没有写完整的 `scores` 或 `probs` 矩阵。为了让逻辑清楚，代码故意写成“一个 CUDA block 处理一行、一个线程做主要计算”的形式；真实高性能实现会把 dot product、max/sum 归约和 `V` 累加拆给一个 warp 或一个 CTA 并行完成。

```cuda
#include <cuda_fp16.h>
#include <math_constants.h>

// Q/K/V/O shape: [B, H, N, D]
// D 和 BK 用模板参数固定，方便编译器把 acc/scores 放进寄存器或本地数组。
template<int D, int BK>
__global__ void fa1_forward_one_row_kernel(
    const half* __restrict__ Q,
    const half* __restrict__ K,
    const half* __restrict__ V,
    half* __restrict__ O,
    int B,
    int H,
    int N,
    float scale,
    bool causal
) {
    int row = blockIdx.x;
    int head = blockIdx.y;
    int batch = blockIdx.z;

    // 教学版只用一个线程说明算法；生产版会用整个 block 协同。
    if (threadIdx.x != 0) return;

    int base = (batch * H + head) * N;
    const half* q = Q + (base + row) * D;

    float m = -CUDART_INF_F;
    float d = 0.0f;
    float acc[D];

    #pragma unroll
    for (int t = 0; t < D; ++t) {
        acc[t] = 0.0f;
    }

    for (int k0 = 0; k0 < N; k0 += BK) {
        float scores[BK];
        float m_local = -CUDART_INF_F;

        #pragma unroll
        for (int j = 0; j < BK; ++j) {
            int col = k0 + j;
            float s = -CUDART_INF_F;

            if (col < N && (!causal || col <= row)) {
                s = 0.0f;
                const half* k = K + (base + col) * D;

                #pragma unroll
                for (int t = 0; t < D; ++t) {
                    s += __half2float(q[t]) * __half2float(k[t]);
                }
                s *= scale;
            }

            scores[j] = s;
            m_local = fmaxf(m_local, s);
        }

        float m_new = fmaxf(m, m_local);
        float old_scale = (m == -CUDART_INF_F) ? 0.0f : expf(m - m_new);

        float d_new = d * old_scale;
        #pragma unroll
        for (int j = 0; j < BK; ++j) {
            d_new += expf(scores[j] - m_new);
        }

        float acc_old_scale = (d_new == 0.0f) ? 0.0f : (d * old_scale) / d_new;
        #pragma unroll
        for (int t = 0; t < D; ++t) {
            acc[t] *= acc_old_scale;
        }

        #pragma unroll
        for (int j = 0; j < BK; ++j) {
            int col = k0 + j;
            if (col >= N || (causal && col > row)) continue;

            float p = expf(scores[j] - m_new) / d_new;
            const half* v = V + (base + col) * D;

            #pragma unroll
            for (int t = 0; t < D; ++t) {
                acc[t] += p * __half2float(v[t]);
            }
        }

        m = m_new;
        d = d_new;
    }

    half* out = O + (base + row) * D;
    #pragma unroll
    for (int t = 0; t < D; ++t) {
        out[t] = __float2half_rn(acc[t]);
    }
}

// 示例 launch:
// dim3 grid(N, H, B);
// fa1_forward_one_row_kernel<64, 64><<<grid, 1>>>(Q, K, V, O, B, H, N, 1.0f / sqrtf(64.0f), true);
```

这段代码最重要的不是性能，而是结构：`scores` 只在当前 `K/V` block 内短暂存在；`m`、`d` 和 `acc` 才是跨 block 保留下来的状态。也就是说，它手工实现的是前面推导的那条在线递推，而不是先调用一个现成 FA kernel。
