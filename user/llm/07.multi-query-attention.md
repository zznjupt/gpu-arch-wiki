### Multi-Query Attention

`Multi-Query Attention` 简写为 `MQA`。它保留多个 query heads，但让所有 query heads 共享同一组 `K/V` heads。

```text
Multi-Head Attention:
Q heads:  H
K heads:  H
V heads:  H

Multi-Query Attention:
Q heads:  H
K heads:  1
V heads:  1
```

直观理解：模型仍然可以从多个角度发问，但大家查的是同一套索引和内容。这样 decode 阶段需要保存和读取的 KV Cache 会大幅减少。

### KV Heads

#### 共享 K/V

标准 MHA 中，第 `h` 个 query head 使用自己的 `K_h/V_h`：

$$
O_h =
\operatorname{softmax}
\left(
\frac{Q_hK_h^T}{\sqrt{d_h}}
\right)V_h
$$

MQA 中，所有 query heads 使用同一组 `K/V`：

$$
O_h =
\operatorname{softmax}
\left(
\frac{Q_hK^T}{\sqrt{d_h}}
\right)V
$$

因此 KV Cache 的 head 数从 `H` 变成 `1`：

$$
M_{KV}^{MQA}
\approx
\frac{1}{H}
M_{KV}^{MHA}
$$

实际节省还会受到实现、数据类型、对齐和其他缓存开销影响，但主趋势就是 `K/V` head 数越少，decode 读写压力越小。

### Decode

#### 为什么推理收益明显

Decode 阶段每次只生成一个 token，但它要读取所有历史 token 的 K/V。序列越长，读取 KV Cache 的带宽压力越大。

MQA 直接减少历史 `K/V` 的体积：

```text
标准 MHA: 每个 query head 读取自己的历史 K/V
MQA:      所有 query heads 读取共享历史 K/V
```

所以 MQA 对长上下文、高并发 batch、decode-heavy 的服务场景特别有用。它减少的是内存流量和缓存占用，而不是把 attention 数学近似成稀疏 attention。

### Tradeoff

#### 质量和效率的平衡

MQA 的代价是表达能力可能下降。因为所有 query heads 共享同一套 `K/V`，不同 head 在“可查询的历史表示”上自由度变少。

实际模型通常需要在训练阶段就使用 MQA，或者从 MHA 模型做额外适配训练。直接把一个已经训练好的 MHA 模型硬改成 MQA，通常不能保证质量。

### Summary

Multi-Query Attention 用多个 `Q` heads 共享一组 `K/V`，显著降低 KV Cache 和 decode 带宽压力。它很适合推理，但效率收益和模型质量之间需要重新训练或适配来平衡。
