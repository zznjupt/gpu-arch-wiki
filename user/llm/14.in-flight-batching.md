### In-flight Batching

`In-flight Batching` 也常叫 `continuous batching` 或动态 batching。它解决的是固定 batch 在 decode 阶段浪费位置的问题。

普通 batching 往往是在请求开始时组一次 batch。In-flight batching 则是在生成过程中持续调整 batch：有请求结束，就把它移出去；有新请求到来，就尽快补进来。

```text
step 1:  A  B  C  D
step 2:  A  B  _  D   C 结束
step 3:  A  B  E  D   新请求 E 进入
```

这样 GPU 不会因为 batch 里有请求提前结束而长期空着位置。

### Decode

#### 为什么 decode 需要动态调度

LLM 的 decode 是逐 token 生成。每一步，batch 里的每个活跃请求通常只生成一个新 token：

```text
active requests
  -> decode one token
  -> update KV Cache
  -> check stop condition
  -> next step
```

不同请求的停止时间不一样。有的很快遇到 EOS，有的还要继续生成很久。固定 batch 会让短请求结束后留下空槽，直到整个 batch 结束才释放调度资源。

In-flight batching 把 batch 看成一个会流动的队列，而不是一个一次性确定的数组。

### Scheduler

#### 每一步重新组织活跃请求

一个 serving scheduler 大致要做这些事：

1. 收集当前还没结束的请求。
2. 为每个请求保留对应的 KV Cache 状态。
3. 在每个 decode step 选择一组活跃请求组成 batch。
4. 把结束的请求移出。
5. 把排队的新请求加入。

```text
waiting queue
      ↓
active batch -> model decode step -> finished requests
      ↑
KV Cache state stays with each request
```

关键点是：请求可以换 batch 位置，但它自己的 KV Cache 必须跟着请求走。PagedAttention 这类 KV Cache 管理方式能让这种动态调度更容易实现。

### Prefill

#### prefill 和 decode 不是同一种负载

In-flight batching 还要处理 prefill 和 decode 混合的问题：

- Prefill：一次处理很多 prompt tokens，计算量大。
- Decode：每个请求每步只生成一个 token，更偏内存带宽和调度。

如果大量 prefill 请求插入得太激进，可能会拉高已有 decode 请求的延迟；如果完全不插入，等待队列吞吐又会变差。实际系统通常会给 prefill 和 decode 设置调度策略，避免互相拖垮。

### Summary

In-flight Batching 把 batch 从“固定一组请求”变成“每一步动态更新的活跃请求集合”。它能显著减少 decode 阶段的空槽浪费，提高 GPU 利用率，但依赖细粒度 scheduler 和稳定的 KV Cache 管理。
