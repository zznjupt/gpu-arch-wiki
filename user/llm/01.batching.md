### Batching

`Batching` 是 LLM 推理里最直接的吞吐优化：一次不要只处理一个请求，而是把多个请求合在一个 batch 里，让 GPU 同时算。

模型权重对所有请求是共享的。如果每次只跑一个请求，GPU 往往会花很多时间等待内存搬运，计算单元吃不满；把多个请求合在一起后，同一轮矩阵乘可以处理更多 token，权重读取也能被更多请求摊薄。

```text
请求 A -> \
请求 B ->  同一个模型 forward
请求 C -> /
```

Batching 的目标不是让单个请求的数学计算变少，而是让同一张 GPU 在单位时间内服务更多请求。

### Static Batching

#### 固定 batch 的问题

最朴素的做法是把若干请求凑成一个固定 batch，然后一起执行。这个思路在 `Prefill` 阶段比较自然，因为每个请求的 prompt 已经给定，可以打包成一次较大的计算。

但到了 `Decode` 阶段，事情会变麻烦。不同请求生成长度不同：

```text
请求 A: 生成 20 tokens 后结束
请求 B: 生成 200 tokens 后结束
请求 C: 生成 80 tokens 后结束
```

如果 batch 固定不变，已经结束的请求会提前离开，剩下的位置不能立刻被新请求补进来。GPU 上的有效 batch size 会越来越小，吞吐也会掉下来。

```text
step 1:  A  B  C  D
step 2:  _  B  C  D
step 3:  _  B  _  D
step 4:  _  B  _  _
```

这也是后面需要 `In-flight Batching` 的原因：LLM serving 里的 batch 不应该只在请求开始时组织一次，而应该在生成过程中持续调整。

### Memory

#### batch size 被 KV Cache 限制

Batch size 变大以后，吞吐通常会更好，但显存压力也会线性增长。主要压力不只来自输入输出 token，还来自每个请求都要维护自己的 KV Cache。

可以粗略理解成：

$$
M_{KV}
\propto
B
\times
L
\times
N_l
\times
H_{kv}
\times
D_h
$$

其中 `B` 是 batch size，`L` 是 sequence length，`N_l` 是层数，`H_{kv}` 是 KV heads 数量，`D_h` 是单个 head 的维度。

所以 batching 的上限经常不是“GPU 算不动”，而是“同时放不下这么多请求的 KV Cache”。长上下文、长输出、多并发请求会一起把显存占满。

### Tradeoff

#### 吞吐和延迟的取舍

Batching 提高的是系统吞吐，但它可能增加单个请求的等待时间：

- 为了凑 batch，新请求可能要等一小段时间。
- batch 太大时，每一步 decode 的执行时间会变长。
- 请求长度差异越大，固定 batch 越容易产生空槽浪费。

实际 serving 系统通常会在延迟 SLA 和吞吐之间设一个平衡点：不要每来一个请求就立刻单独跑，也不要为了追求大 batch 让请求排队太久。

### Summary

Batching 的核心是把多个请求合在一起，让 GPU 的矩阵计算更饱满。它是提升吞吐的基础手段，但会受到 KV Cache 显存和请求长度差异的限制。普通固定 batch 解决不了 decode 阶段动态进出的请求，所以推理服务里通常还需要更细的调度方式，也就是 `In-flight Batching`。
