### Pipeline Parallelism

`Pipeline Parallelism` 把一个大模型按层切开，放到多张 GPU 上。每张 GPU 只保存其中一段连续层，输入从第一段开始，算完后把中间激活传给下一张 GPU。

```text
tokens
  -> GPU 0: layer 0..11
  -> GPU 1: layer 12..23
  -> GPU 2: layer 24..35
  -> GPU 3: layer 36..47
  -> logits
```

它解决的是单张 GPU 放不下完整模型的问题。每张卡只承担一部分层，模型参数和中间激活都被拆到了多张卡上。

### Pipeline

#### 像流水线一样推进

如果一次只处理一个 batch，pipeline parallelism 会有明显空闲：

```text
时间 1: GPU0 工作, GPU1 空闲, GPU2 空闲
时间 2: GPU0 空闲, GPU1 工作, GPU2 空闲
时间 3: GPU0 空闲, GPU1 空闲, GPU2 工作
```

为了让多张 GPU 同时忙起来，系统会把大 batch 拆成多个 `microbatch`，让它们像流水线一样前后相接。

```text
时间 1: GPU0 处理 microbatch 0
时间 2: GPU0 处理 microbatch 1, GPU1 处理 microbatch 0
时间 3: GPU0 处理 microbatch 2, GPU1 处理 microbatch 1, GPU2 处理 microbatch 0
```

这样能提高多 GPU 利用率，但 pipeline 两端仍然会有填充和排空阶段，这部分空闲通常叫 `pipeline bubble`。

### Inference

#### 对推理的影响

在 LLM 推理里，pipeline parallelism 常见于模型太大、单卡放不下的情况。它的好处是可以横向扩展模型容量；代价是每个 token 都要跨 GPU 传递中间激活。

Prefill 阶段通常 batch 和矩阵都比较大，pipeline 更容易吃满。Decode 阶段每次只生成少量新 token，单步计算更小，跨设备通信和 pipeline bubble 更容易变成瓶颈。

### Tradeoff

Pipeline parallelism 的关键取舍是：

- 模型可以被切到多张 GPU 上，单卡显存压力下降。
- 层与层之间需要传递激活，跨 GPU 通信变多。
- microbatch 能缓解空闲，但不能完全消除 pipeline bubble。
- 对 decode 这种小步长、自回归的阶段，调度难度更高。

### Summary

Pipeline Parallelism 是按层切模型。它让超大模型可以放进多张 GPU，但性能取决于流水线是否足够饱满，以及层间通信和 bubble 是否可控。
